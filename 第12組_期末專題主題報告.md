## 一、專題概述（Planning）

### 1.1 動機與背景
**說明為什麼選擇這個主題，這個問題在生活中或產業中真實存在嗎？**

在日常生活中，我們需要搭乘大眾運輸工具時，每次在連接，都會花費大量時間，像是我搭乘捷運要轉公車，每次都無法很準確的定位公車或該運輸工具的定位，因此，我們想透過這樣的 app 節省時間，並了解大眾運輸動線。

### 1.2 目標使用者
**這個系統是為誰設計的？他們有什麼特性或需求？**

台中市民，或來台中需要搭乘大眾運輸的民眾。他們需求是要有準確的到站時間，不只是單單只有時刻表而已，用以節省不必要的等候時間，透過這款 app 可以讓使用者不必為了等不到公車而焦慮，對於行程的規劃也能夠更加的明確。

### 1.3 專題目標
**完成後，這個系統能解決什麼問題？請用條列方式列出 3–5 點。**

- 目標一：時程準確性
- 目標二：大眾運輸工具檢視圖
- 目標三：定位精準化
- 目標四：規劃路線與替代方案
- 目標五：價格推算
- 目標六：在台中地區規劃
- 目標七：到達目的地所需時間

## 二、需求分析（Requirements Analysis）

### 2.1 功能性需求（Functional Requirements）
**系統「必須能做到」的事情。請列出核心功能（至少 3 項）。**

| 功能編號 | 功能名稱 | 功能描述 |
| --- | --- | --- |
| F-01 | 一站式平台 | 整合所有大眾運輸資訊 |
| F-02 | 即時動態資訊 | 串接 API 提供公車到站即時位置、捷運班次 |
| F-03 | 轉乘時間最佳化路徑演算 | 輸入起終點後，系統會自動計算結合「捷運 + 公車」、「火車 + YouBike」 等組合的最佳路徑，並提供預計抵達時間 |
| F-04 | 試算時間與價格功能 | 給予使用者選擇的空間 |
| F-05 | 常用路線與轉乘點收藏 | 提供常用路線「一鍵收藏」 |
| F-06 | 多運具即時動態整合看板 | 在地圖選取特定轉乘點時，系統需同時呈現「公車進站倒數」與「捷運發車時刻」的整合清單，而非切換選單。 |
| F-07 | 站點擁擠度 | 顯示站點人潮擁擠程度 |

### 2.2 非功能性需求（Non-functional Requirements）
**系統的品質要求，例如效能、安全性、易用性等。**

| 類別 | 需求描述 |
| --- | --- |
| 到站提醒 | 像是捷運、火車、公車，在到站前三分鐘做一個提醒，會在手機螢幕跳出通知 |
| 準確度 | 讓公車上的 gps 定位以及上面的 wifi 都能夠順暢地傳達到我們做的公車 app 上面，和及時路線地圖，讓使用者可明確知道目前公車所在的位置，以及移動 |
| 易用性 | 以簡單的公車號碼查詢就能得知，公車要來的時間，就可以做推算 |
| 站點收費 | 依據所在地最近的站牌，到每站點的價格(其中包括捷運、公車、火車) |
| 導航指引 | 配合地圖，導航至所要去的站點，不必再開啟地圖查詢是去程還是返程的站牌 |

### 2.3 範圍限制（Out of Scope）
**明確說明這次專題「不會」做的功能，避免範圍膨脹。**

- 本次不包含：離線地圖導航模式
- 本次不包含：行動支付與線上購票（系統僅提供交通資訊查詢，不包含實體刷卡、電子票證如悠遊卡儲值或 App 內掃碼支付功能。）
- 本次不包含：跨縣市長途導航
- 本次不包含：使用者帳號與會員系統
